{{ page.title }}

  • 代码逻辑应当直截了当，叫缺陷难以隐藏； 尽量减少依赖关系，使之便于维护； 依据某种分层战略完善错误处理代码； 性能调至最优，省得引诱别人做没规矩的优化，搞出一堆混乱来。 整洁的代码只做好一件事。
  • 糟糕的代码想做太多事，它意图混乱、目的含混。 整洁的代码力求集中。 每个函数、每个类和每个模块都全神贯注于一事，完全不受四周细节的干扰和污染。
  • (代码应该是易读的,容易理解的)整洁的代码简单直接。整洁的代码如同优美的散文。整洁的代码从不隐藏设计者的意图，充满了 干净利落的抽象和直截了当的控制语句。
  • 整洁的代码应可由作者之外的开发者阅读和增补。(后面读代码的人加上备注,如果作者自己都读不懂代码,那么说明这段代码不好?) 它应当有单元测试和验收测试。 它使用有意义的命名。 它只提供一种而非多种做一件事的途径。 它只有尽量少的依赖关系，而且要明确地定义和提供清晰、尽量少的 API。 代码应通过其字面表达含义，因为不同的语言导致并非所有必需信息均可通过代码自身清晰表达。
  • 整洁的代码便于其他人加以增补。这看似显而易见，但亦不可过分强调。毕竟易读的代码和易修改的代码之间还是有区别的。(???)
  • 没有测试的代码不干净。(!!!???) 代码块越小越好。

  • 贝克的简单代码规则,依其重要顺序： — 能通过所有测试； — 没有重复代码； — 体现系统中的全部设计理念； — 包括尽量少的实体，比如类、方法、函数等。
  • 消除重复和提高表达力。 有意义的命名是体现表达力的一种方式。 如果对象功能太多，最好是切分为两个或多个对象。 减少重复代码，提高表达力，提早构建简单抽象。
